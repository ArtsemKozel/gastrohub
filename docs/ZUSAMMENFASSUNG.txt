 WAS IST GASTROHUB?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

GastroHub ist eine Neuentwicklung auf Basis von PlanIt. PlanIt war eine
monolithische Single-Page-App (eine große admin.js + employee.js). GastroHub
zerlegt diese Logik in eine modulare Ordnerstruktur mit getrennten JS-Dateien
pro Feature-Bereich und drei klar getrennten Nutzergruppen.

Dieses Repo (gastrohub-new) ist die Basic-Variante für Ein-Filial-Betriebe.
Die Mehr-Filialen-Variante ist gastrohub-iaro (Supabase: ecxvbjrxhyllqnuknwly,
Repo: https://github.com/ArtsemKozel/gastrohub-iaro).

  Neu in GastroHub:
    gh_time_entries       Stempelzeiten der Mitarbeiter
                          Spalten: id, user_id, employee_id,
                          clock_in (timestamptz), clock_out (timestamptz),
                          note (text),
                          is_manual (boolean) — true wenn manuell nachgetragen
                          Hinweis: RLS auf anon key blockiert chained
                          .insert().select() — nach INSERT immer separates
                          SELECT durchführen

    gh_breaks             Pausen zu einem Stempelzeit-Eintrag
                          Spalten: id, user_id, employee_id,
                          time_entry_id (FK → gh_time_entries.id),
                          break_start (timestamptz), break_end (timestamptz)
                          RLS-Policies: SELECT, INSERT, UPDATE, DELETE
                          Hinweis: Beim Bearbeiten einer Stempelzeit wird der
                          bestehende gh_breaks-Eintrag gelöscht und neu
                          angelegt. Die Pausendauer wird als synthetischer
                          Eintrag gespeichert: break_start = clock_in,
                          break_end = clock_in + pauseMin Minuten

    planit_payrolls       Vorlohnabrechnungen
                          Spalten: id, user_id, employee_id, month (YYYY-MM),
                          data (jsonb), created_at

    planit_terminations   Kündigungsanträge
                          Spalten: id, user_id, employee_id, requested_date,
                          approved_date, reason, status, initiated_by, pdf_url

    employment_phases     Beschäftigungsphasen
                          Spalten: id, employee_id, start_date, end_date

    temperature_devices   Temperaturkontrolle-Geräte
                          Spalten: id, user_id, name, min_temp, max_temp,
                          created_at (Sortierung: immer ascending)

    temperature_logs      Temperaturmesswerte
                          Spalten: id, user_id, employee_id, device_id,
                          temperature, note, log_date, created_at

    planit_hour_balances  Stundenkonto-Saldi
                          Enthält carry_over_minutes für Monatsübertrag.
                          Auto-Übertrag: loadAdminStunden() berechnet
                          Vormonat-Saldo automatisch wenn kein manueller
                          Eintrag existiert und der Vormonat Daten hat.

  Infrastruktur: OneSignal SDK (Mitarbeiter-App) + Cloudflare Worker als
  Relay (gastrohub-notify.artsem86.workers.dev)

  Funktion sendPushNotification(title, message, external_id?) in notify.js
  sendet einen POST an den Worker. external_id ist die employee.id als String
  (OneSignal external_id) — bei Aufruf ohne external_id wird an alle gesendet.

  Ausgelöste Benachrichtigungen:
    Schichtplan veröffentlicht   → an alle Mitarbeiter (kein external_id)
    Offene Schicht zugewiesen    → an den zugewiesenen Mitarbeiter
    Andere Bewerber abgelehnt    → an alle anderen Bewerber derselben Schicht
    Urlaubsantrag genehmigt      → an den Mitarbeiter
    Urlaubsantrag abgelehnt      → an den Mitarbeiter
    Kündigung genehmigt          → an den Mitarbeiter
    Kündigung abgelehnt          → an den Mitarbeiter

Aufträge werden als zusammenhängender Fließtext auf Deutsch formuliert —
keine Aufzählungslisten, keine Codeblöcke im Auftragstext. Der Auftragstext
nennt immer den Dateinamen und den Funktionsnamen, der geändert werden soll.

Pro Auftrag wird genau eine inhaltliche Änderung beschrieben. Mehrere
unabhängige Änderungen werden als separate Aufträge gestellt.

Mobile und Desktop werden immer gleichzeitig berücksichtigt.

Nach jedem abgeschlossenen Feature wird sofort committed und gepusht.
Nach jedem Commit wird sofort ohne Nachfrage gepusht.

 PRE-COMMIT-HOOK (Cache-Busting)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Ein Pre-Commit-Hook aktualisiert automatisch die ?v=-Query-Parameter aller
<script src="...?v=...">-Tags in admin.html, employee.html und pos.html.
Dadurch werden Browser-Caches bei jedem Deploy invalidiert.
